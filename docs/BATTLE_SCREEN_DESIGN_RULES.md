# BATTLE_SCREEN_DESIGN_RULES

この文書は、戦闘画面を場当たりに作らないための設計ルールである。
キャラクター別コマンド入力、キャンセル、対象選択、呪文選択、道具選択、浮動ログ小窓を前提に、スマホ縦画面で破綻しにくい完成形から逆算する。

関連文書:

- `docs/SCREEN_FLOW_RULES.md`
- `docs/UI_STYLE_RULES.md`
- `docs/COMBAT_RULES.md`
- `docs/PARAMETER_DESIGN_RULES.md`
- `docs/WIZARDRY_REFERENCE_NOTES.md`

## 1. 今回固定する前提

- 戦闘は専用ウィンドウで扱う。
- 戦闘中は通常探索操作を受け付けない。
- 戦闘ログは、戦闘画面の通常縦積みレイアウトに参加させない。
- 戦闘ログは独立した浮動小窓として扱う。
- 戦闘コマンドは、最終的にキャラクター別入力にする。
- キャラクター別入力では、戻る、キャンセル、対象選択、呪文選択、道具選択が必ず発生する。
- v23系の全体コマンド方式は、検証用の暫定構造であり、完成形ではない。
- 画面上部の大見出し枠は置かない。
- コマンドを右カラムへ押し込まない。
- ラベルと値は近接させる。
- ボタンや行が画面外へ押し出される構造にしない。

## 2. 戦闘の完成形から見た基本フロー

最終的な戦闘入力は以下の流れを前提とする。

```text
敵が現れる
  ↓
戦闘開始ログ
  ↓
1人目の行動入力
  ↓
必要なら対象選択 / 呪文選択 / 道具選択
  ↓
2人目の行動入力
  ↓
...
  ↓
全員分の行動入力完了
  ↓
行動確認
  ↓
ラウンド実行
  ↓
ログ表示
  ↓
敵全滅 / パーティ全滅 / 逃走 / 次ラウンド
```

この流れにより、戦闘画面は常に以下の情報を扱える必要がある。

- 現在入力中のキャラクター。
- そのキャラクターが前衛か後衛か。
- そのキャラクターが選択可能なコマンド。
- すでに行動決定済みのキャラクター。
- 現在がコマンド選択中か、対象選択中か、呪文選択中か、道具選択中か。
- 戻る操作でどの状態へ戻るか。
- ログ表示中か、入力可能か。
- ラウンド実行中か、次入力待ちか。

## 3. 画面を構成する4層

戦闘画面は、通常フロー3層と浮動ログ1層に分ける。

```text
通常フロー:
1. 敵表示エリア
2. パーティ状態エリア
3. 入力エリア

浮動レイヤー:
4. 戦闘ログ小窓
```

禁止する構造:

```text
敵表示
LOG
パーティ状態
コマンド
```

この構造は、LOGが通常レイアウトを押し下げるため採用しない。

採用する構造:

```text
戦闘ウィンドウ
  ├─ 敵表示エリア
  ├─ パーティ状態エリア
  ├─ 入力エリア
  └─ 浮動LOG小窓
```

LOG小窓は絶対配置または同等の浮動配置にし、敵表示、パーティ状態、入力エリアの高さ計算に参加させない。

## 4. 戦闘画面のレイアウト方針

### 4.1 全体の優先順位

スマホ縦画面では、以下を優先する。

1. 現在の敵が分かる。
2. パーティの危険状態が分かる。
3. 現在誰の入力中か分かる。
4. 選べるコマンドが押しやすい。
5. 戻る・キャンセルが迷わない。
6. 戦闘ログが読めるが、操作領域を押し下げない。

### 4.2 推奨基本構造

```text
┌────────────────────┐
│ 敵表示エリア        │
│ 敵画像 / 敵グループ │
└────────────────────┘
┌────────────────────┐
│ パーティ状態        │
│ 6人分のHP/AC/ST     │
└────────────────────┘
┌────────────────────┐
│ 入力エリア          │
│ 現在キャラ + コマンド│
└────────────────────┘

    ┌ 浮動LOG小窓 ───┐
    │ 最新ログ本文    │
    └───────────────┘
```

LOGは見た目としては別ウィンドウだが、戦闘専用ウィンドウの内側に浮かせる。
ブラウザ全体に対するグローバルトーストにはしない。

## 5. 敵表示エリア

### 5.1 目的

敵表示エリアは、敵の存在と数を把握する場所である。
長い説明や戦闘ログは置かない。

### 5.2 表示要素

- 低解像度敵画像。
- 敵グループ名または仮分類名。
- 出現数。
- デバッグ用HP表示は検証用であり、本番表示として固定しない。

### 5.3 サイズ方針

- 敵表示エリアは画面上部に置く。
- 高さは固定寄りにする。
- 敵画像が大きすぎてパーティ状態や入力エリアを圧迫しないようにする。
- 複数敵の場合は、敵画像を横並びまたは圧縮表示する。
- 敵の数が増えた場合、画像をすべて大きく表示しようとしない。

## 6. パーティ状態エリア

### 6.1 目的

パーティ状態エリアは、戦闘判断に必要な最低限の状態を表示する。
キャラクター詳細画面の代替にしない。

### 6.2 表示項目

戦闘中は以下を優先する。

- 名前または短縮名。
- HP。
- AC。
- 状態。
- 入力済みかどうか。
- 現在入力中かどうか。

職業は、必要なら短縮表示に留める。
呪文回数や所持品一覧はここに出さない。

### 6.3 レイアウト候補

6人を2列3行で表示する。
前衛3人、後衛3人の区別を残す。

```text
前: ADEL  HP34 AC4 OK  入力中
前: BERN  HP29 AC5 OK  未
前: CEL   HP21 AC6 OK  未
後: DANA  HP18 AC8 OK  未
後: ERI   HP16 AC9 OK  未
後: FAY   HP20 AC7 OK  未
```

または、狭い場合は以下のように1行を圧縮する。

```text
ADEL 34/34 AC4 OK ▶
BERN 29/29 AC5 OK -
```

### 6.4 禁止事項

- 表ヘッダが大きすぎて本文を圧迫する構造にしない。
- ラベルと数値を大きく離さない。
- パーティ状態欄に長い職業名、説明文、呪文回数、所持品を入れない。
- 入力エリアやLOGを押し出すほど高さを増やさない。

## 7. 入力エリア

### 7.1 役割

入力エリアは、現在の戦闘状態に応じて内容が変わる差し替え領域である。
右カラム固定ではなく、画面下部の横幅を使う。

### 7.2 入力エリアの状態

入力エリアは以下の状態を持つ。

- `character_command`: キャラクター別コマンド選択。
- `target_select`: 攻撃・呪文・道具の対象選択。
- `spell_select`: 呪文選択。
- `item_select`: 道具選択。
- `round_confirm`: 全員入力後の確認。
- `round_resolving`: ラウンド実行中。
- `battle_result`: 勝利・敗北・逃走などの結果表示。

### 7.3 現在キャラクター表示

入力エリアの上部には、現在入力中のキャラクターを短く表示する。

```text
入力: ADEL / FIG / 前衛
```

長い大見出しにしない。
必要なのは、今誰の番かだけである。

### 7.4 コマンド表示方針

コマンドは縦リストを基本とする。
ただし、行高と文字サイズを調整し、画面下部に収める。

候補:

```text
▶ 攻撃
  守る
  呪文
  道具
  逃げる
  戻る
```

ボタン式にする場合も、視覚上は縦リストとして読む。
横2列のボタン群を標準にはしない。

### 7.5 コマンド数が多い場合

コマンド数が増えても、入力エリア全体が画面外へ伸びないようにする。

- 入力エリアの最大高さを決める。
- コマンドリストのみスクロール可能にする。
- 重要コマンドを上に置く。
- 使用不可コマンドは並べて説明に使わない。
- 使用不可理由は必要な時だけLOGまたは短い補助表示に出す。

## 8. キャラクター別コマンド定義

ここでのコマンド名は機能ラベルであり、最終表示名ではない。
日本語表示名は後で調整する。

### 8.1 基本コマンド候補

- `attack`: 攻撃。
- `defend`: 守る。
- `spell`: 呪文。
- `item`: 道具。
- `run`: 逃げる。
- `back`: 戻る。

### 8.2 前衛キャラクター

前衛は物理攻撃を標準で選べる。

```text
攻撃
守る
道具
逃げる
戻る
```

呪文使用可能職の場合は `呪文` も加える。

### 8.3 後衛キャラクター

後衛の攻撃は、武器射程や仕様に応じて有効/無効を決める。
初期実装では、後衛の物理攻撃を無効にするか、威力や対象を制限する余地を残す。

```text
攻撃   ※条件付き
守る
呪文   ※使用可能職のみ
道具
逃げる
戻る
```

### 8.4 呪文使用可能キャラクター

呪文使用可能職は、呪文コマンドを持つ。
呪文残回数がない場合は、コマンドを非表示または無効にする。

無効ボタンを大量に並べて状態説明にしない。

### 8.5 道具コマンド

道具は、キャラクターが所持する所持品インスタンスから選ぶ。
アイテムはスタックしないため、同じアイテムが複数ある場合も別行として扱う。

道具選択画面では、表示可能数を超えた場合にリスト内スクロールにする。
入力エリア全体を押し下げない。

### 8.6 逃げるコマンド

逃げるはパーティ全体に影響する行動として扱う。
キャラクター別入力の中で選ばれた場合、そのラウンドの逃走方針として扱うか、即時確認へ進むかを後続で決める。

初期実装では、`逃げる` を選んだ時点で確認状態に移す案が安全。

## 9. キャンセル・戻る設計

### 9.1 必須方針

キャラクター別入力では、キャンセルが必須である。
戻る操作なしにキャラクター別コマンドを実装しない。

### 9.2 戻るの種類

戻るには複数の意味がある。

- 対象選択から、そのキャラクターのコマンド選択へ戻る。
- 呪文選択から、そのキャラクターのコマンド選択へ戻る。
- 道具選択から、そのキャラクターのコマンド選択へ戻る。
- 現在キャラクターの行動選択を取り消し、前のキャラクターへ戻る。
- 全員入力後の確認から、最後のキャラクター選択へ戻る。

### 9.3 行動キュー

各キャラクターの選択済み行動は、行動キューに保存する。

```json
[
  {
    "characterId": "adel",
    "command": "attack",
    "target": "enemy_inst_0001"
  },
  {
    "characterId": "bern",
    "command": "defend",
    "target": null
  }
]
```

戻る場合は、行動キューの最後の要素を削除し、そのキャラクターを再入力対象にする。

### 9.4 画面表示上の入力済み表示

パーティ状態エリアでは、各キャラクターが以下のどれかを分かるようにする。

- 現在入力中。
- 入力済み。
- 未入力。
- 行動不能。

記号例:

- `▶`: 現在入力中。
- `✓`: 入力済み。
- `-`: 未入力。
- `×`: 行動不能。

記号は仮であり、最終表示は後で調整する。

## 10. 対象選択

### 10.1 対象選択の前提

攻撃、攻撃呪文、道具使用では対象選択が発生する。
対象選択は入力エリアを差し替えて表示する。

### 10.2 敵対象選択

敵が複数グループいる場合、まずグループを選ばせる。
単体指定が必要な場合は、次に個体を選ばせる。

初期実装ではグループ単位でよい。

```text
対象を選択
▶ 粘性体 x2
  虫型 x3
  戻る
```

### 10.3 味方対象選択

回復や補助の場合、味方を選ぶ。

```text
対象を選択
▶ ADEL 34/34
  BERN 18/29
  戻る
```

### 10.4 戻る

対象選択の戻るは、そのキャラクターのコマンド選択へ戻る。
前のキャラクターへ戻る操作とは区別する。

## 11. 呪文選択

### 11.1 呪文選択の構造

呪文は、系統とレベル別使用回数制を前提にする。
MP制にはしない。

表示候補:

```text
呪文を選択: ADEL
MAGE 1: 仮呪文A  残 2
MAGE 1: 仮呪文B  残 2
戻る
```

### 11.2 表示方針

- 使用可能な呪文だけを優先表示する。
- 使えない呪文を大量に並べない。
- 呪文名は仮名扱い。
- 長い説明文は表示しない。
- 必要ならLOG小窓に「使えない。」など短く表示する。

### 11.3 呪文レベル切り替え

呪文数が増えたら、レベル選択を一段挟む。

```text
呪文レベル
MAGE 1 残2
MAGE 2 残0
PRIEST 1 残1
戻る
```

初期実装では、使用可能呪文一覧だけでもよい。
ただし、将来的にレベル選択へ分割できる構造にする。

## 12. 道具選択

### 12.1 道具選択の構造

道具選択は、キャラクターの所持品インスタンスを表示する。

```text
道具を選択: ADEL
▶ 薬
  松明
  戻る
```

### 12.2 スクロール

所持品が増えた場合、道具リストだけをスクロールさせる。
入力エリア全体や戦闘ウィンドウ全体を縦に押し広げない。

### 12.3 戻る

道具選択の戻るは、そのキャラクターのコマンド選択へ戻る。

## 13. 全員入力後の確認

全員分の行動が決まったら、即時ラウンド実行せず、確認状態を挟む余地を残す。

```text
行動を開始しますか
▶ 決定
  戻る
```

初期実装では確認を省略してもよいが、設計上は `round_confirm` 状態を持つ。
戻るを選ぶと最後のキャラクターの入力へ戻る。

## 14. ラウンド実行中

ラウンド実行中は、キャラクター別入力を受け付けない。

- LOG文字送り中は、コマンド入力よりログ送りを優先する。
- LOG表示中に入力された場合、まず全文表示する。
- 全ログが表示されてから次ラウンドの入力へ移る。

## 15. 浮動LOG小窓

### 15.1 基本方針

LOGは通常フローの1段ではない。
独立した小窓として、戦闘ウィンドウ内に浮かせる。

### 15.2 表示位置

推奨位置は、入力エリア直上に重なる位置。
ただし、入力エリアを押し下げない。

```text
敵表示
パーティ状態
[浮動LOG小窓]
入力エリア
```

LOGは視覚的には入力エリアの直上に見えるが、DOM/CSS上は通常フローに参加しない。

### 15.3 表示仕様

- 半透明黒背景。
- 白枠または細い枠線。
- 高さ固定。
- 最大2行程度。
- 最新ログを主表示。
- 古いログは薄く1行まで、または表示しない。
- 文字送り速度は内部定数で調整可能。
- 表示完了後、一定時間で薄くする余地を残す。

### 15.4 タップ仕様

LOG小窓全体をタップ対象にする。

- 文字送り中: 即時全文表示。
- 全文表示済みで次ログあり: 次ログへ進む。
- 全文表示済みで次ログなし: そのまま残すか薄くする。

iPhone Safariでの実機確認が必要。
モックDOMでイベントハンドラが呼ばれても、実機確認済みとは言わない。

### 15.5 コマンド入力との関係

- LOG表示中にコマンドを押した場合、まずLOG全文表示を優先する。
- その押下では行動を進めない。
- LOGが完了している場合だけ、コマンド入力を受け付ける。

## 16. 戦闘画面で表示しないもの

戦闘画面には以下を常時表示しない。

- 大きな `ENCOUNTER` 見出し。
- 長い説明文。
- キャラクター詳細画面相当の能力値一覧。
- 呪文レベル別全回数一覧。
- 所持品全リスト。
- 敵の全パラメータ。
- デバッグ用HP表示の本番固定。

## 17. 実装段階

### 17.1 v23eで行うこと

- この設計に従って、戦闘画面の構造を再構築する。
- LOGを通常フローから外し、浮動LOG小窓にする。
- 入力エリアをキャラクター別コマンド前提の構造にする。
- ただし、v23eでは全コマンドの本処理を実装しない。
- まずは1人ずつ入力が進む骨格、戻る、行動キュー、ラウンド実行の最小形を作る。

### 17.2 v24以降で行うこと

- 攻撃対象選択の本実装。
- 呪文選択の本実装。
- 道具選択の本実装。
- 敵からの反撃。
- 逃走判定。
- 経験値・戦利品。
- 宝テーブル接続。

## 18. 検査観点

### 18.1 構造検査

- LOGが通常縦積みレイアウトの1段として存在していないこと。
- LOGが敵表示、パーティ状態、入力エリアを押し下げていないこと。
- 戦闘画面に大見出し枠がないこと。
- 入力エリアが右カラム固定ではないこと。
- キャラクター別入力状態が存在すること。
- 行動キューが存在すること。
- 戻る操作で行動キューを戻せること。

### 18.2 動作検査

- 戦闘開始時に1人目が入力対象になる。
- コマンド選択後、必要に応じて対象選択へ進む。
- 戻るで現在キャラのコマンド選択へ戻れる。
- 2人目以降へ進める。
- 前のキャラへ戻れる。
- 全員入力後にラウンド実行へ進める。
- ラウンド実行中はコマンド入力を受け付けない。
- LOG表示中はLOG送りが優先される。

### 18.3 実機未確認として扱う項目

こちらではiPhone Safari実機確認ができないため、以下は未確認として報告する。

- LOG小窓タップで全文表示できるか。
- 入力エリアのボタンが実機で押しやすいか。
- コマンドリストが画面下部に収まるか。
- 対象選択や道具選択のスクロールが自然か。
- 文字送り速度の体感。

## 19. 禁止事項

- キャラクター別入力を前提にせず、全体コマンドだけを基準にレイアウトしない。
- キャンセルや戻るを後付けにしない。
- LOGを通常レイアウト内の段として置かない。
- コマンドを右カラムへ押し込まない。
- 大見出しで戦闘画面の縦幅を消費しない。
- ボタンやリストが画面外へ押し出される構造にしない。
- モックDOMでの確認をiPhone Safari実機確認済みと書かない。
- 敵名、呪文名、アイテム名をAI案だけで正本化しない。
